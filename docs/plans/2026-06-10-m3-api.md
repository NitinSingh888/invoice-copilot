# Milestone 3 — FastAPI API Layer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. TDD, `- [ ]` steps, each task green (`pytest`+`ruff`+`mypy src`) and committed. Commits author as NitinSingh888 — no Co-Authored-By trailer.

**Goal:** Expose the M2 services over a versioned FastAPI HTTP API (`/api/v1`): invoices (create→process, list, get, action), rules (list, propose, activate, enable/disable), audit (trail + chain verify), health — plus an app factory that serves the `web/` prototype as static files. (The conversational `chat` endpoint and LLM extraction are M4; M3 accepts already-structured invoice data, which is also how the deterministic demo/tests drive it.)

**Architecture:** `api/v1/routes/*` (thin HTTP handlers) → `services/*` (M2 orchestration) → `repositories`/`domain`. Pydantic v2 DTOs in `schemas/`. DB session + role injected via FastAPI dependencies; tests override the session dependency with an in-memory SQLite session.

**Tech Stack:** FastAPI, Starlette `StaticFiles`, `httpx`/`fastapi.testclient`. Add deps: `fastapi`, `uvicorn[standard]`, `python-multipart` (uploads later), and dev dep `httpx`.

**Spec:** `docs/specs/2026-06-09-ap-agent-design.md` §9 (API/tools), §12 (api layer), §13 (UI consumes these).

### Task 2: Invoice routes (create→process, list, get, action)
**Files:** `api/v1/routes/invoices.py`; register in `router.py`; test `tests/integration/api/test_invoices.py`; add a small seed helper `tests/integration/api/_seed.py`.

- [ ] `_seed.py`: helper to seed an approved vendor + a PO + N prior cleared invoices into a given Session (so auto-clear is reachable), reused by tests.
- [ ] `invoices.py` router (all depend on `get_db`, `get_role`):
  - `POST ""` body `InvoiceIn` → build `InvoiceData` (generate id if absent, e.g. `inv-<hex8>`), call `pipeline_service.process_invoice(db, invoice_data, confidence)`, then read the persisted invoice; return `ProcessResultOut` (201) with verdict/route/reason/status/findings.
  - `GET ""` → `[InvoiceOut]` from `invoice_repo.list_all`.
  - `GET "/{invoice_id}"` → `InvoiceOut` or raise `NotFoundError`.
  - `POST "/{invoice_id}/action"` body `ActionIn`, role from `get_role`: 
    - If the invoice was escalated and the human action differs from "approve" (i.e. route/hold) OR it's an Acme-style override, call `learning_service.record_correction(...)` capturing the salient warn finding code (recompute findings or store from process step — simplest: re-run enrichment+policy to get the current salient warn finding code; if none, use "MANUAL"). Then call `execution_service.execute(db, invoice_id, action, actor=role, **edit_fields)`. For `edit`, pass `amount`/`route` as fields. Return `InvoiceOut`.
- [ ] Register the invoices router in `api/v1/router.py` under prefix `/invoices`.
- [ ] Tests (`TestClient` with `app.dependency_overrides[get_db]` → in-memory seeded session): 
  - POST a clean low-risk invoice (seeded approved vendor past cold-start, matched PO within tol) → 201, `verdict == "AUTO_CLEAR"`, `status == "queued"`.
  - POST a 13%-over-PO invoice → `verdict == "ESCALATE"`, `status == "needs"`.
  - GET list returns both; GET one by id; GET missing id → 404.
  - POST `/{id}/action` `{action:"route"}` with header `X-Role: priya` on an escalated invoice → 200, status `"routed"`; an audit event with actor `user:priya` (or `priya`) exists.
- [ ] Commit: `feat(api): invoice routes (process, list, get, action)`.

### Task 3: Rules + Audit routes
**Files:** `api/v1/routes/rules.py`, `audit.py`; register; tests `test_rules.py`, `test_audit.py`.

- [ ] `rules.py`:
  - `GET ""` → `[RuleOut]` from `rule_repo.list_all`.
  - `POST "/propose"` → `learning_service.propose_rule(db)`; if None → 204 No Content; else `RuleProposalOut`.
  - `POST "/activate"` body `ActivateRuleIn` → recompute `propose_rule(db)`; if None → 409; else `learning_service.activate_rule(db, proposal=..., threshold_pct=body.threshold_pct, route=body.route)` → `RuleOut` (201).
  - `PATCH "/{rule_id}"` body `RuleStatusIn` → `learning_service.set_rule_status(db, rule_id, status)` → `RuleOut` (404 if missing).
- [ ] `audit.py`:
  - `GET "/{invoice_id}"` → `AuditTrailOut(events=audit_service.trail(db, invoice_id), chain_verified=audit_service.verify(db))`.
- [ ] Register both routers (`/rules`, `/audit`).
- [ ] Tests: seed 3 consistent corrections → `POST /rules/propose` returns proposal with `threshold_pct=="0.08"`; `POST /rules/activate {threshold_pct:"0.08",route:"Priya"}` → 201 active rule; `GET /rules` lists it; `PATCH /rules/{id} {status:"disabled"}` → status disabled. Process an invoice then `GET /audit/{id}` → events list non-empty and `chain_verified is True`.
