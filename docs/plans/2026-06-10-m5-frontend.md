# Milestone 5 — Frontend Live-Mode Wiring Implementation Plan

> **For agentic workers:** superpowers:subagent-driven-development. Backend tasks are TDD; frontend tasks are verified by running the app in a browser (no unit-test framework for the Babel-standalone prototype). Commits author as NitinSingh888 — no Co-Authored-By trailer.

**Goal:** Make the `web/` prototype work against the real API in a **Live mode**, while keeping the existing **Demo mode** (the flake-proof scripted 7-beat walkthrough) as the default. Add **seed data** so the live backend boots with the demo batch (and so the Render deploy seeds-on-boot).

**Architecture:** Backend gains `app/seed.py` (idempotent seed of vendors/POs/a demo invoice batch in `received` status, mirroring `web/data.js`) invoked from the app lifespan when the DB is empty, plus a `POST /api/v1/demo/reset` endpoint. Frontend gains `web/api.js` (fetch wrappers) and a `Demo | Live` toggle: in **Live** the existing components (Queue, ApprovalCard, AuditDrawer, RulesDrawer, Composer) are driven by real API calls; in **Demo** the current scripted machine runs unchanged.

**Spec:** §2 (the demo), §13 (UI), §10/§11 (mock-safe, real backend). Demo mode = choreographed; Live mode = real backend, functional (not choreographed).

### Task 1: Seed data (backend, TDD)
**Files:** `src/app/seed.py`; modify `src/app/main.py` (lifespan seed-if-empty) and add a demo-reset route `src/app/api/v1/routes/demo.py`; tests `tests/unit/test_seed.py`, `tests/integration/api/test_demo_reset.py`.

- [ ] Read `web/data.js` to mirror the demo dataset. `seed.py`:
  - `SEED_VENDORS`: "Acme Corp" (approved, default_approver "Priya") + the other design vendors (approved).
  - `SEED_POS`: POs for the matched invoices. For the routine/auto-clear invoices, PO amount == invoice amount (within tolerance). For the 3 Acme escalations, PO amounts that put them +4/+6/+7% over (from data.js `poAmount`). For INV-4471 (Northwind) a PO making it ~13% over.
  - `SEED_INVOICES`: the ~16-invoice batch in status `received` with vendor/amount/po_number/invoice_number from data.js (the routine ones, the 4 escalations, the Cyberdyne injection one with no PO + unknown vendor, the Stark exact-duplicate — for which ALSO seed a prior `cleared` invoice with the same vendor+invoice_number).
  - Also seed enough prior `cleared` invoices per approved vendor to satisfy `cold_start_n` so routine ones auto-clear.
  - `def seed(s, *, force: bool = False) -> None`: if `force`, delete all rows first; if the DB already has invoices and not force, no-op (idempotent). Insert vendors, POs, prior-cleared, then the batch.
  - `def is_empty(s) -> bool`.
- [ ] `main.py` lifespan: after `init_db`, open a Session and `seed(s)` if empty.
- [ ] `demo.py`: `POST "/demo/reset"` → `seed(db, force=True)` → `{"status": "reseeded", "invoices": <count>}`. Register at `/demo`.
- [ ] Tests: `seed(s)` populates vendors/POs/invoices; calling twice is idempotent (no duplicates); `force=True` resets; after seed, processing a routine invoice auto-clears and the duplicate one blocks (sanity). `POST /api/v1/demo/reset` → 200 with a count; the queue (`GET /invoices`) is non-empty after.
- [ ] Commit: `feat(seed): demo dataset + seed-on-boot + /demo/reset`.

### Task 2: API client + Live mode (frontend)
**Files:** `web/api.js`; modify `web/index.html` (load api.js), `web/app.jsx`, `web/panels.jsx` (TopBar toggle). No unit tests (prototype) — verified in Task 3.

- [ ] `web/api.js`: a small `window.IC_API` with `async` fetch wrappers (base `/api/v1`): `listInvoices()`, `process(invoiceIn)`, `getInvoice(id)`, `action(id, body, role)`, `chat(message, history, role)`, `listRules()`, `proposeRule()`, `activateRule(body)`, `setRuleStatus(id, status)`, `trail(id)`, `reset()`. JSON in/out; send `X-Role` header.
- [ ] `index.html`: add `<script src="api.js"></script>` before the babel components.
- [ ] `app.jsx`: add `const [mode, setMode] = React.useState("demo")`. Default **demo** (current scripted behavior unchanged). When `mode === "live"`:
  - On switch to live: call `reset()` then `listInvoices()` → populate `invoices` from real rows (map API InvoiceOut → the shape the Queue expects: id, vendor, amount, po, status, findings as needed). Clear scripted messages; show a short "Live mode — connected to the API" system message.
  - Composer `handleSend` → `IC_API.chat(text, history, role)`; render the agent `reply` as an agent message; if the result includes processed counts, refresh the queue via `listInvoices()`; if a card/escalation is relevant, render it from real data.
  - Approval card actions → `IC_API.action(invId, {action, ...}, role)` then refresh queue + (optionally) the resolved summary.
  - Audit drawer (`openTrail`) → `IC_API.trail(invId)` → render the real event stream (map fields to the existing AuditDrawer shape; show `chain verified`).
  - Rules drawer → `IC_API.listRules()`.
  - Keep all Demo-mode code paths intact (guard every live branch behind `mode === "live"`).
- [ ] `panels.jsx` TopBar: add a `Demo | Live` segmented toggle (next to the Play button) calling `onMode`. Wire `onMode`/`mode` props from App.
- [ ] Commit: `feat(web): API client + Live mode wired to the real backend`.
